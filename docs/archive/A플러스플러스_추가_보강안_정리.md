# A++ 추가 보강안 정리

이 문서는 기존 탈중앙화 커뮤니티 설계의 큰 틀 위에 추가로 확정한 **A++ 보강안**만 정리한 문서입니다.

핵심 방향은 다음과 같습니다.

> **발견은 안전하게, 이미지는 분리해서, 관리는 쉽게.**

A++는 공식 서버나 공식 카테고리 목록 없이도 사용자가 쉽게 커뮤니티에 들어오고, 관련 커뮤니티를 찾고, 이미지를 볼 수 있게 하되, 운영 책임과 위험은 각 역할별로 분리하는 구조입니다.

---

## 1. A++의 핵심 목표

A++의 목표는 세 가지입니다.

```text
1. 유저 친화성
- 초대 링크나 QR로 쉽게 들어오기
- 관련 커뮤니티 쉽게 찾기
- 이미지 쉽게 보기
- 신고 쉽게 하기

2. 관리자 친화성
- 카테고리 번들 쉽게 만들기
- 신고 큐 쉽게 처리하기
- 미디어 노드 쉽게 운영하기
- 문제 발생 시 비상 모드로 빠르게 대응하기

3. 책임 분리
- 공식 서버 없음
- 공식 카테고리 목록 없음
- 공식 미디어 노드 없음
- 이미지 저장은 별도 media-node가 담당
- 카테고리 관리자는 표시/숨김/정책 판단을 담당
```

A++는 기술적으로는 분산형이지만, 일반 사용자가 느끼는 경험은 최대한 일반 커뮤니티처럼 단순해야 합니다.

---

## 2. 첫 진입: 서명된 카테고리 번들

공식 카테고리 목록은 만들지 않습니다.
대신 사용자는 다음 방식으로 커뮤니티에 들어옵니다.

```text
- 초대 링크
- QR 코드
- 카테고리 번들 파일
- 친구가 공유한 커뮤니티 번들
- 커뮤니티가 자체 배포하는 번들
```

### 2.1 카테고리 번들에 포함되는 정보

```text
- category_id
- 표시 이름
- 카테고리 헌장
- 텍스트 노드 목록
- 허용 미디어 노드 목록
- 관리자 공개키 목록
- 신고 정책
- 항소 정책
- 번들 생성 시각
- 번들 만료 시각
- 번들 서명
```

### 2.2 유저 화면 예시

```text
game-korea 카테고리를 추가하시겠습니까?

포함된 항목:
- 텍스트 노드 3개
- 미디어 노드 2개
- 관리자 5명
- 카테고리 헌장 있음
- 신고/항소 정책 있음

안전 확인:
- 번들 서명 확인됨
- 만료되지 않음
- 미디어 노드 정책 확인됨

[추가] [자세히 보기] [취소]
```

### 2.3 관리자 기능

관리자는 버튼으로 번들을 만들 수 있어야 합니다.

```text
[초대 링크 만들기]
[QR 코드 만들기]
[카테고리 번들 내보내기]
```

### 2.4 설계 원칙

```text
- 사용자는 복잡한 키와 해시를 몰라도 됨
- 클라이언트가 번들 서명과 만료 여부를 자동 확인
- 고급 정보는 “자세히 보기”에 숨김
- 공식 추천이 아니라 사용자가 받은 번들을 추가하는 구조
```

## 3. 카테고리 상태 5단계

카테고리 상태는 너무 복잡하게 만들지 않고 5개만 둡니다.

```text
active
정상 운영

safe_mode
스팸, 신고 급증, 미디어 노드 문제 등으로 보수적 운영

frozen
장악 의심, 관리자 분쟁, 헌장 변경 위험으로 거버넌스 일시 정지

recovery
관리자 키 탈취, 세력 장악, 대규모 공격 이후 복구 절차

deprecated
새 카테고리로 이주 권장
```

### 3.1 유저 화면 예시

```text
이 카테고리는 현재 안전 모드입니다.

일시적으로:
- 이미지 자동 로드 꺼짐
- 신규 미디어 노드 추가 제한
- 신고 많은 글 자동 숨김
- 헌장 변경 투표 중지

사유: 미디어 노드 신고 급증
```

### 3.2 관리자 화면 예시

```text
카테고리 상태 변경:

[정상 운영]
[안전 모드]
[동결]
[복구]
[이주 권장]
```

### 3.3 설계 원칙

```text
- 유저에게는 현재 상태와 제한 사유를 쉽게 설명
- 관리자는 버튼 중심으로 상태 변경
- safe_mode와 recovery는 관리자·유저 모두 이해하기 쉬워야 함
```

## 4. 관련 커뮤니티 찾기

기술 문서에서는 이 기능을 **주변 노드 발견**이라고 부를 수 있습니다.
하지만 유저 UI에서는 더 쉬운 이름인 **관련 커뮤니티 찾기**를 사용합니다.

목적은 공식 추천 없이도 사용자가 다른 커뮤니티를 찾게 하는 것입니다.

### 4.1 핵심 원칙

```text
- 개인별 가입 목록 공개 금지
- opt-in 기반
- 민감 카테고리 기본 제외
- 정확한 숫자 숨김
- 최소 인원 기준 적용
- 조작 방지 기준 적용
- 공식 추천 서버 없음
```

### 4.2 유저 화면 예시

```text
관련 커뮤니티 찾기

이 카테고리 이용자들이 함께 많이 보는 커뮤니티입니다.
개별 사용자의 가입 목록은 공개되지 않습니다.

1. game-meme
   함께 보는 사람 규모: 많음
   현재 카테고리와 겹침: 높음
   내가 신뢰한 사용자: 여러 명
   [추가] [자세히]

2. retro-game
   함께 보는 사람 규모: 보통
   현재 카테고리와 겹침: 보통
   [추가] [자세히]
```

## 5. 관련 커뮤니티 표시 기준

“많음/보통/적음”은 감이 아니라 고정 수치로 산정합니다.

정확한 실제 인원수는 프라이버시 때문에 숨기지만, 등급 기준은 공개하고 고정합니다.

---

### 5.1 유효 참여자 조건

관련 커뮤니티 발견에 반영되는 사용자는 아래 조건을 만족해야 합니다.

```text
유효 참여자 조건:

1. 사용자가 관련 커뮤니티 발견 참여를 직접 허용함
2. 계정 생성 후 14일 이상 지남
3. 현재 카테고리에서 최근 30일 안에 활동함
4. 정상 활동이 5회 이상 있음
5. 스팸·도배·신고 남용 제한 중이 아님
6. 민감 카테고리는 사용자가 명시적으로 공개 허용한 경우만 포함
```

---

### 5.2 기본 산정 기준

두 가지 값을 봅니다.

```text
C = 함께 이용하는 유효 참여자 수
R = 현재 카테고리 유효 참여자 중 함께 이용하는 비율
```

기본 노출 조건은 다음입니다.

```text
C ≥ 30명
R ≥ 3%
```

즉, 30명 미만이거나 3% 미만이면 관련 커뮤니티 목록에 표시하지 않습니다.

---

### 5.3 함께 보는 사람 규모

| 표시 | 기준 |
|---|---:|
| 적음 | 30~59명 |
| 보통 | 60~149명 |
| 많음 | 150명 이상 |

---

### 5.4 현재 카테고리와의 겹침

| 표시 | 기준 |
|---|---:|
| 낮음 | 3~6.9% |
| 보통 | 7~14.9% |
| 높음 | 15% 이상 |

---

### 5.5 예시

```text
game-meme
- 함께 보는 사람 규모: 많음
- 현재 카테고리와 겹침: 높음
```

```text
general-chat
- 함께 보는 사람 규모: 많음
- 현재 카테고리와 겹침: 낮음
```

이렇게 하면 “사람이 많은 커뮤니티”와 “현재 카테고리와 진짜 가까운 커뮤니티”를 구분할 수 있습니다.

## 6. 민감 카테고리 기준

정치, 내부고발, 상담, 소수자, 청소년, 지역 분쟁 같은 민감 카테고리는 더 보수적으로 처리합니다.

```text
민감 카테고리 기본값:
관련 커뮤니티 찾기 off
```

카테고리 헌장으로 켜더라도 기준을 높입니다.

```text
민감 카테고리 노출 조건:

C ≥ 100명
R ≥ 10%
정확한 숫자 표시 금지
개인별 목록 표시 금지
```

표시도 약하게 합니다.

```text
관련 커뮤니티 있음
```

즉, 민감 카테고리에서는 “많음/보통/적음”도 최소화합니다.

---

## 7. 내가 신뢰한 사용자 기준

공개 집계와 별개로, 내 클라이언트는 내가 신뢰한 사용자 기준으로 로컬 추천을 만들 수 있습니다.

```text
내 신뢰 그래프 기준:

1~2명:
표시 안 함

3~4명:
내가 신뢰한 일부 사용자가 이용

5~9명:
내가 신뢰한 여러 사용자가 이용

10명 이상:
내가 신뢰한 사용자가 많이 이용
```

이 값은 서버가 계산하지 않고 **내 클라이언트가 로컬에서 계산**합니다.

## 8. 관리자 설정 프리셋

관리자가 숫자를 마음대로 세세하게 조절하면 혼란이 생깁니다.
따라서 관리자에게는 프리셋만 제공합니다.

```text
관련 커뮤니티 발견 설정

○ 꺼짐

○ 기본
   - C ≥ 30
   - R ≥ 3%

○ 엄격
   - C ≥ 60
   - R ≥ 7%

○ 민감 보호
   - C ≥ 100
   - R ≥ 10%
   - 정확한 등급 표시 없음
```

권장 기본값은 다음입니다.

```text
일반 카테고리:
기본

정치/상담/민감 카테고리:
꺼짐 또는 민감 보호
```

## 9. 미디어 노드: 별도 프로그램으로 분리

클라이언트는 이미지를 볼 수 있게 합니다.
하지만 이미지 저장·업로드·썸네일·신고·캐시삭제는 별도 **media-node 프로그램**이 담당합니다.

```text
client:
- 일반 사용자용
- 글/댓글/투표/이미지 보기
- media_ref 해석
- 이미지 신고/숨김 버튼
- 기본적으로 미디어 서버 기능 없음

text-node:
- 게시글, 댓글, 투표, 헌장 전파
- 이미지 원본 저장 안 함

media-node:
- 이미지 저장 전용 프로그램
- 개인/집단/카테고리가 선택적으로 운영

protocol:
- 공통 객체 스키마
- 서명, 해시, media_ref, tombstone 정의
```

## 10. media-node 원클릭 운영

미디어 노드는 개인이나 집단이 쉽게 운영할 수 있어야 합니다.

배포 방식은 세 가지입니다.

```text
1. Docker Compose
서버 운영자용

2. 단일 실행 파일
개인 운영자용

3. 로컬 웹 대시보드
비개발자 관리자용
```

---

### 10.1 media-node 기본값

```text
media-node 기본값:

- 영상 업로드 off
- 파일 업로드 off
- 공개 검색 off
- 이미지 최대 2MB
- JPG/PNG/WebP만 허용
- SVG 금지
- EXIF 제거 on
- TTL 30일
- 저장공간 상한 필수
- 일일 업로드 제한 on
- 신고함 on
- 캐시삭제 지원 on
```

## 11. media-node 비상 모드

미디어 노드 운영자가 공격이나 신고 폭주를 받았을 때 버튼 하나로 방어할 수 있어야 합니다.

```text
비상 모드:

- 새 업로드 중단
- 신고 누적 이미지 자동 숨김
- 외부 카테고리 요청 차단
- 원본 대신 썸네일만 제공
- 특정 카테고리 media_ref 제공 중지
- 저장공간 초과 시 오래된 이미지부터 만료
```

### 11.1 미디어 노드 대시보드 예시

```text
미디어 노드 대시보드

상태:
- 저장공간: 12GB / 50GB
- 오늘 업로드: 231개
- 신고 대기: 7개
- 차단된 이미지: 41개

빠른 조치:
[새 업로드 일시 중지]
[신고 많은 이미지 숨김]
[특정 카테고리 차단]
[비상 모드 켜기]
```

## 12. media-node manifest

각 미디어 노드는 자기 정책을 공개해야 합니다.
카테고리는 이 manifest를 보고 미디어 노드 허용 여부를 투표합니다.

```json
{
  "type": "media_node_manifest",
  "node_pubkey": "media-node-pubkey-1",
  "name": "game-meme-media-node",
  "operator_type": "individual_or_group",
  "accepted_media": ["image/webp", "image/jpeg", "image/png"],
  "max_file_size": "2MB",
  "video_allowed": false,
  "default_ttl": "30d",
  "exif_strip": true,
  "public_search": false,
  "report_supported": true,
  "cache_delete_supported": true,
  "emergency_mode_supported": true,
  "allowed_categories": ["game-korea", "meme"],
  "policy_url": "optional",
  "compliance_statement": "operator_responsibility"
}
```

## 14. 공개 로그와 비공개 증거 분리

관리 투명성은 필요하지만, 민감한 원본은 공개하면 안 됩니다.

```text
공개 로그:
- 조치 시각
- 조치 유형
- 사유 코드
- 대상 해시
```

## 15. Tombstone 표현

프로토콜 내부에서는 `tombstone`을 사용합니다.
하지만 유저에게는 어려운 용어를 쓰지 않습니다.

```text
프로토콜:
tombstone

관리자 UI:
전파 중단 기록

유저 UI:
숨김 기록
```

범위는 세 가지로 단순화합니다.

```text
local
내 클라이언트에서만 숨김

category
이 카테고리 피드에서 숨김

node
이 노드가 전파/캐시 제공 중단
```

중요한 점은 다음입니다.

```text
전역 삭제 명령이 아님
각 카테고리/노드 범위의 표시 제한·전파 중단 기록임
```

```text
이미지 보안 기본값:
```

## 17. 릴리즈 서명

악성 클라이언트가 퍼지면 전체 구조가 무너집니다.

```text
업데이트 정책:

- GitHub 릴리즈 확인 가능
- 자동 업데이트 기본 off
- 릴리즈 파일 서명 필수
- 체크섬 제공
- 클라이언트가 서명 검증
- 서명 실패 시 설치 차단
```

유저 화면은 단순하게 갑니다.

```text
새 버전이 있습니다.

버전: 0.2.0
서명: 확인됨
변경사항: 보안 수정

[업데이트] [나중에]
```

## 19. A++ 최종 구조

```text
A++ 구조:

1. 공식 서버 없음
2. 공식 카테고리 목록 없음
3. 첫 진입은 서명된 초대 링크/QR/카테고리 번들
4. 카테고리 상태 5단계
   active / safe_mode / frozen / recovery / deprecated
5. 관련 커뮤니티 찾기
   opt-in 익명 집계 기반
6. 관련 커뮤니티 표시 기준 고정
   C ≥ 30, R ≥ 3%
7. 민감 카테고리는 기본 off 또는 강화 기준
   C ≥ 100, R ≥ 10%
8. 클라이언트는 이미지 보기 지원
9. 이미지 저장은 별도 media-node 프로그램
10. media-node 원클릭 운영 지원
11. media-node 비상 모드 제공
12. media-node manifest 공개
13. 신고는 쉬운 UI + 내부 표준 코드
14. 공개 로그와 비공개 증거 분리
15. tombstone은 UI에서 숨김/전파중단 기록으로 표현
16. 클라이언트 이미지 보안 기본 적용
17. 릴리즈 서명 필수
18. MVP 제외 항목 명확화
```

## 20. 한 문장 요약

> **공식 서버 없이 초대 번들로 들어오고, 관련 커뮤니티는 opt-in 익명 집계로 찾으며, 이미지는 클라이언트에서 보되 저장은 별도 media-node가 맡고, 신고·숨김·비상 대응은 유저와 관리자가 쉽게 처리할 수 있게 만든 구조입니다.**

더 짧게 말하면:

> **발견은 안전하게, 이미지는 분리해서, 관리는 쉽게.**
