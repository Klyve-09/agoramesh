# A+++ 추가 보강안 정리

## 0. 문서 목적

이 문서는 기존 A++ 설계 위에 추가로 채택할 보강안을 정리한 문서입니다.

기존 큰 방향은 유지합니다.

- 중앙 서버 없음
- 공식 카테고리 목록 없음
- 공식 검색 인덱서 없음
- 공식 게이트웨이 없음
- 공식 미디어 노드 없음
- 공식 추천 알고리즘 없음
- 클라이언트와 프로토콜 중심
- 카테고리별 자치
- 이미지 저장은 별도 `media-node`로 분리

A+++의 핵심은 기능을 더 많이 붙이는 것이 아니라, **유저와 관리자가 실제로 덜 힘들게 쓸 수 있게 만드는 것**입니다.

> **A+++의 핵심: 운영 피로도 줄이기 + 안전한 온보딩 + 쉬운 이주**

---

## 1. A+++ 최종 채택 항목

이번에 채택하는 핵심 보강안은 6개입니다.

```text
1. 신고 자동 묶음
2. 카테고리 생성 마법사
3. 카테고리 건강 상태 + safe_mode 추천
4. 미디어 노드 운영 점검표
5. 사용자 데이터 백업/이주
6. 처음 보는 것 경고 체계
```

이 6개는 모두 **유저 친화성**과 **관리자 친화성**을 우선해서 고른 항목입니다.

---

# 2. 신고 자동 묶음

## 2.1 왜 필요한가

신고 기능이 생기면 관리자는 곧바로 신고 큐에 시달리게 됩니다.

특히 다음 상황이 자주 생깁니다.

```text
- 같은 게시물에 신고가 여러 번 들어옴
- 같은 이미지에 신고가 몰림
- 같은 유저가 도배해서 신고가 반복됨
- 같은 미디어 노드가 문제를 일으킴
- 신고 테러로 큐가 폭주함
```

관리자가 신고를 하나씩 확인하게 만들면 금방 지칩니다.

따라서 신고는 **개별 신고 나열**이 아니라 **자동 묶음**으로 보여줘야 합니다.

---

## 2.2 자동 묶음 기준

신고는 다음 기준으로 묶습니다.

```text
신고 자동 묶음 기준:

- 같은 게시물 신고 묶기
- 같은 댓글 신고 묶기
- 같은 이미지/media_ref 신고 묶기
- 같은 유저에 대한 반복 신고 묶기
- 같은 미디어 노드 관련 신고 묶기
- 같은 사유 코드 신고 묶기
- 같은 시간대에 발생한 대량 신고 묶기
```

---

## 2.3 관리자 화면 예시

```text
신고 큐

[긴급 묶음]

1. 같은 이미지에 대한 신고 12건
   사유: 불법 성적 이미지 의심
   대상: media_ref ...
   관련 카테고리: game-korea

   빠른 조치:
   [미디어 숨김]
   [미디어 노드 임시차단]
   [기각]
   [자세히 보기]

2. 같은 유저에 대한 신고 18건
   사유: 도배/스팸
   최근 1시간 게시물: 47개

   빠른 조치:
   [24시간 작성 제한]
   [게시물 일괄 숨김]
   [기각]
   [자세히 보기]


[일반 묶음]

3. 카테고리 규칙 위반 신고 31건
   [일괄 검토]
   [나중에]
```

---

## 2.4 처리 원칙

```text
유저 친화:
- 신고 후 처리 상태를 알 수 있음
- 같은 내용을 반복 신고하지 않아도 됨

관리자 친화:
- 같은 문제를 한 번에 처리 가능
- 신고 폭주 상황에서 큐가 덜 복잡함
- 긴급 신고와 일반 신고가 분리됨

남용 방지:
- 신규 계정 신고 가중치 낮춤
- 신고 남용자는 신고 영향도 제한
- 신고 묶음도 공개 로그에 요약 기록
```

# 3. 카테고리 생성 마법사

## 3.1 왜 필요한가

카테고리에는 헌장, 관리자, 투표권 조건, 미디어 정책, 신고 정책, 포크 절차 등이 필요합니다.

하지만 일반 사용자가 처음부터 긴 설정 파일이나 JSON을 직접 작성하게 하면 카테고리를 만들기 어렵습니다.

그래서 카테고리 생성은 **마법사 방식**으로 만들어야 합니다.

---

## 3.2 생성 흐름

```text
카테고리 만들기

1단계: 기본 정보
2단계: 운영 방식
3단계: 미디어 정책
4단계: 관리자 시작 방식
5단계: 관련 커뮤니티 발견 정책
6단계: 초대 번들 생성
```

---

## 3.3 1단계: 기본 정보

```text
카테고리 이름:
game-korea

설명:
한국 게임 잡담 커뮤니티

주제 태그:
게임, 잡담, 밈
```

내부적으로는 표시 이름과 고유 ID를 분리합니다.

```text
display_name:
game-korea

category_id:
생성자 공개키 + 이름 + 생성 시각 + 헌장 해시
```

---

## 3.4 2단계: 운영 방식

관리자가 어려운 권한 설정을 직접 하나하나 고르지 않도록 프리셋을 제공합니다.

```text
운영 방식 선택

○ 가벼운 관리
  - 스팸 숨김
  - 신고 큐 정리
  - 미디어 임시 숨김
  - 계정 제한 없음

○ 일반 관리
  - 24시간 임시 숨김
  - 미디어 노드 임시 차단
  - 작성 제한 제안
  - 공개 로그 필수

○ 엄격한 관리
  - 72시간 임시 숨김
  - 반복 위반 계정 작성 제한
  - 미디어 노드 긴급 차단
  - 항소 절차 필수
```

기본 추천은 **일반 관리**입니다.

---

## 3.5 3단계: 미디어 정책

```text
미디어 정책 선택

○ 이미지 없음
○ 외부 링크만 허용
○ 신뢰 미디어 노드만 허용
○ 자체 미디어 노드 사용
```

기본 추천은 다음입니다.

```text
일반 카테고리:
신뢰 미디어 노드만 허용

법률/상담/민감 카테고리:
이미지 없음 또는 외부 링크만 허용
```

---

## 3.6 4단계: 관리자 시작 방식

```text
관리자 시작 방식

○ 임시 관리자 1명으로 시작
○ 초기 관리자 3명으로 시작
○ 첫 7일 후 선거
○ 첫 30명 참여 후 선거
```

기본 추천은 **초기 관리자 3명 + 첫 7일 후 재신임**입니다.

---

## 3.7 5단계: 관련 커뮤니티 발견 정책

```text
관련 커뮤니티 찾기 설정

○ 꺼짐
○ 기본
   - C ≥ 30
   - R ≥ 3%

○ 엄격
   - C ≥ 60
   - R ≥ 7%

○ 민감 보호
   - C ≥ 100
   - R ≥ 10%
   - 정확한 등급 표시 없음
```

기본 추천은 다음입니다.

```text
일반 카테고리:
기본

정치/상담/청소년/소수자/내부고발:
꺼짐 또는 민감 보호
```

---

## 3.8 6단계: 초대 번들 생성

카테고리 생성이 끝나면 바로 초대 수단을 만들 수 있어야 합니다.

```text
카테고리가 생성되었습니다.

초대 방법:
[초대 링크 만들기]
[QR 코드 만들기]
[카테고리 번들 파일 내보내기]
```

# 4. 카테고리 건강 상태 + safe_mode 추천

## 4.1 왜 필요한가

카테고리는 어느 날 갑자기 무너지지 않습니다.

보통 다음 신호가 먼저 나타납니다.

```text
- 신고 급증
- 신규 계정 급증
- 미디어 노드 장애
- 관리자 응답 지연
- 장기 활동자 이탈
- 포크 논의 증가
- 스팸 증가
```

이 신호를 유저와 관리자가 보기 쉽게 보여주면, 문제가 커지기 전에 대응할 수 있습니다.

---

## 4.2 카테고리 건강 상태 지표

```text
카테고리 건강 상태 지표:

- 활동량 정상 여부
- 신규 계정 급증 여부
- 신고 급증 여부
- 관리자 응답 지연 여부
- 미디어 노드 장애 여부
- 포크/이주 논의 증가 여부
- 장기 활동자 이탈 여부
- 스팸 숨김 증가 여부
```

---

## 4.3 유저 화면 예시

정상 상태:

```text
이 카테고리 상태: 안정적

- 활동량 정상
- 신고 급증 없음
- 미디어 노드 정상
```

주의 상태:

```text
이 카테고리 상태: 주의 필요

- 신고가 평소보다 많습니다.
- 미디어 노드 1개가 응답하지 않습니다.
- 관리자 응답이 지연되고 있습니다.
```

---

## 4.4 관리자 대시보드 예시

```text
관리자 대시보드

최근 24시간:
- 신고: 82건, 평소 대비 +230%
- 신규 계정 작성 비율: 41%
- 숨김 처리 대기: 17건
- media-node-a 응답 실패율: 36%
- 장기 활동자 이탈 신호: 중간
```

---

## 4.5 safe_mode 자동 추천 기준

safe_mode는 자동으로 강제 전환하지 않고, 기본적으로 **관리자에게 추천**합니다.

아래 조건 중 2개 이상 발생하면 safe_mode 전환을 권장합니다.

```text
safe_mode 추천 조건:

- 신고량이 평소의 3배 이상
- 신규 계정 작성 비율이 40% 이상
- 특정 media_ref 신고 10건 이상
- 미디어 노드 장애율 30% 이상
- 스팸 숨김이 1시간 내 20건 이상
- 관리자 키 이상 활동 감지
```

관리자 화면:

```text
safe_mode 전환을 권장합니다.

감지된 문제:
- 신고량 급증
- 신규 계정 작성 비율 급증
- 특정 미디어 신고 집중

[24시간 safe_mode 켜기]
[무시]
[자세히 보기]
```

---

## 4.6 safe_mode 적용 시 동작

```text
safe_mode에서 적용되는 조치:

- 이미지 자동 로드 중지
- 처음 보는 미디어 노드 흐림 처리
- 신고 많은 글 자동 숨김 강화
- 신규 미디어 노드 추가 제한
- 헌장 변경 투표 일시 중지
- 신규 계정 작성 제한 강화
- 관련 커뮤니티 찾기 일시 제한
```

# 5. 미디어 노드 운영 점검표

## 5.1 왜 필요한가

미디어 노드는 이미지 저장·썸네일·신고·캐시삭제를 담당하는 별도 프로그램입니다.

하지만 개인이나 작은 집단이 운영하는 경우 설정을 잘못할 수 있습니다.

그래서 대시보드에 **운영 점검표**를 넣습니다.

---

## 5.2 필수 점검 항목

```text
미디어 노드 운영 점검표

필수:
☑ 저장공간 상한 설정됨
☑ 신고함 활성화됨
☑ 캐시삭제 지원 켜짐
☑ EXIF 제거 켜짐
☑ 영상 업로드 꺼짐
☑ 파일 업로드 꺼짐
☑ SVG 업로드 꺼짐
☑ 공개 검색 꺼짐
☑ TTL 설정됨
☑ 일일 업로드 제한 설정됨
```

---

## 5.3 권장 점검 항목

```text
권장:
☐ 운영 정책 문서 연결
☐ 비상 연락 경로 설정
☐ 백업 노드 설정
☐ 카테고리별 업로드 제한 설정
☐ 관리자 2인 이상 설정
☐ 정기 로그 정리 설정
```

---

## 5.4 대시보드 표시 예시

```text
미디어 노드 점검 상태: 양호

필수 항목: 10/10 완료
권장 항목: 3/6 완료

권장 조치:
- 운영 정책 문서를 연결해 주세요.
- 비상 연락 경로를 설정해 주세요.
```

# 6. 사용자 데이터 백업/이주

## 6.1 왜 필요한가

분산 커뮤니티에서는 사용자가 직접 자기 데이터를 관리해야 합니다.

특히 다음 상황에서 백업과 이주가 필요합니다.

```text
- 기기를 바꿈
- 클라이언트를 재설치함
- 카테고리가 포크됨
- 기존 카테고리가 deprecated 상태가 됨
- 미디어 노드 또는 텍스트 노드가 사라짐
- 키를 교체해야 함
```

포크와 이주 구조가 있어도 사용자의 설정을 옮길 수 없으면 실제로는 따라가기 어렵습니다.

---

## 6.2 백업 대상

```text
백업할 것:

- 구독한 카테고리 목록
- 카테고리 번들 목록
- 신뢰한 사용자 목록
- 차단한 사용자 목록
- 차단한 카테고리 목록
- 차단한 미디어 노드 목록
- 관련 커뮤니티 발견 참여 설정
- 이미지 자동 로드 설정
- 신고/숨김 로컬 설정
- 사용자 프로필
- 선택적으로 개인 키
```

---

## 6.3 백업 UI 예시

```text
내 데이터 내보내기

포함 항목:
☑ 구독 카테고리
☑ 차단 목록
☑ 신뢰 목록
☑ 이미지 표시 설정
☑ 관련 커뮤니티 찾기 설정
☐ 개인 키 포함

[암호 걸고 내보내기]
```

---

## 6.4 이주 UI 예시

```text
이 카테고리는 새 카테고리로 이주 중입니다.

기존:
game-korea

이주 후보:
game-korea-classic

내가 신뢰한 사용자 중 여러 명이 새 카테고리를 사용 중입니다.

[기존 유지]
[둘 다 구독]
[새 카테고리로 이동]
```

---

## 6.5 개인 키 백업 원칙

개인 키 백업은 매우 민감하므로 기본으로 포함하지 않습니다.

```text
개인 키 백업 원칙:

- 기본값은 개인 키 제외
- 사용자가 명시적으로 선택해야 포함
- 반드시 암호화
- 복구 문구 또는 백업 파일 제공
- 백업 파일 생성 시 강한 경고 표시
```

# 7. 처음 보는 것 경고 체계

## 7.1 왜 필요한가

공식 서버와 공식 카테고리 목록이 없기 때문에, 사용자는 초대 링크나 번들을 통해 여러 노드에 들어갑니다.

이때 다음 위험이 있습니다.

```text
- 악성 카테고리 번들
- 피싱 노드
- 가짜 미디어 노드
- 갑자기 바뀐 관리자 키
- 갑자기 바뀐 헌장
- 악성 이미지 노드
```

따라서 클라이언트는 사용자가 처음 보는 대상이나 갑자기 바뀐 대상을 자동으로 경고해야 합니다.

---

## 7.2 경고 대상

```text
처음 보는 대상:

- 처음 보는 카테고리
- 처음 보는 텍스트 노드
- 처음 보는 미디어 노드
- 처음 보는 관리자 키
- 처음 보는 카테고리 번들 서명자
- 처음 보는 짤팩
```

```text
갑자기 바뀐 대상:

- 기존과 달라진 관리자 목록
- 기존과 달라진 헌장 해시
- 기존과 달라진 미디어 정책
- 기존과 달라진 텍스트 노드 목록
- 기존과 달라진 미디어 노드 목록
```

---

## 7.3 유저 경고 UI 예시

처음 보는 미디어 노드:

```text
주의: 처음 보는 미디어 노드입니다.

이 노드는 아직 사용자가 신뢰한 적이 없습니다.
이미지는 흐림 처리로 표시됩니다.

[한 번만 보기]
[이 노드 신뢰]
[차단]
```

카테고리 번들 변경:

```text
주의: 이 카테고리 번들의 관리자 목록이 이전과 다릅니다.

변경 내용:
- 관리자 2명 제거
- 관리자 3명 추가

[변경 내용 보기]
[추가 중단]
```

서명 없는 번들:

```text
주의: 이 카테고리 번들은 서명이 확인되지 않았습니다.

공격자가 만든 가짜 번들일 수 있습니다.

[취소]
[위험을 이해하고 계속]
```

---

## 7.4 경고 원칙

```text
경고 원칙:

- 유저에게 해시와 키를 강제로 읽게 하지 않음
- “처음 보는 것”과 “갑자기 바뀐 것”을 쉽게 알려줌
- 기본 선택지는 안전한 쪽으로 둠
- 고급 정보는 자세히 보기에서 제공
```

# 9. 최종 A+++ 구조

A+++까지 반영한 최종 구조는 다음과 같습니다.

```text
A+++ 구조:

1. 공식 서버 없음
2. 공식 카테고리 목록 없음
3. 첫 진입은 서명된 초대 링크/QR/카테고리 번들
4. 관련 커뮤니티 찾기는 opt-in 익명 집계 기반
5. 카테고리 상태는 active / safe_mode / frozen / recovery / deprecated
6. 신고 큐는 자동 묶음 처리
7. 카테고리 생성은 마법사 방식
8. 카테고리 건강 상태 표시
9. 조건 충족 시 safe_mode 전환 추천
10. 클라이언트는 이미지 보기 지원
11. 이미지 저장은 별도 media-node 담당
12. media-node는 원클릭 운영과 비상 모드 지원
13. media-node 운영 점검표 제공
14. 유저 데이터 백업/이주 지원
15. 처음 보는 번들/노드/미디어 노드/관리자 키 경고
16. 공개 로그와 비공개 증거 분리
17. tombstone은 UI에서 숨김/전파중단 기록으로 표현
18. 릴리즈 서명 필수
19. MVP 제외 항목 명확화
```

# 10. 한 문장 요약

A+++는 이렇게 정리할 수 있습니다.

> **관리자가 덜 지치게 신고를 묶고, 유저가 쉽게 카테고리를 만들게 마법사를 넣고, 문제가 커지기 전에 건강 상태와 safe_mode 추천을 보여주고, 미디어 노드 운영자가 실수하지 않게 점검표를 넣고, 유저가 포크·기기변경 때 따라갈 수 있게 백업/이주를 넣고, 처음 보는 번들·노드·미디어 노드는 경고한다.**

더 짧게 말하면:

> **A+++의 핵심은 운영 피로도 줄이기, 안전한 온보딩, 쉬운 이주입니다.**
