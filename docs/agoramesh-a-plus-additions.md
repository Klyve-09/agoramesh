# 추가 보강안 A+ 정리

> 범위: 이 문서는 기존의 큰 틀인 **“중앙 운영자 없는 분산 자치 커뮤니티 프로토콜”** 설명을 반복하지 않고, 이후 대화에서 추가로 보강한 부분만 정리한 문서입니다.

---

## 0. 추가 보강안의 핵심

기존 기본안은 다음과 같이 **A+ 버전**으로 확장합니다.

```text
기본안 A+ =
초대 기반 진입
+ 주변 노드 발견
+ 카테고리 자치
+ 클라이언트 이미지 보기
+ 별도 media-node 프로그램
+ 안전장치 강화
```

한 문장으로 정리하면 다음과 같습니다.

> 공식 서버나 공식 카테고리 목록 없이도 사용자가 커뮤니티를 찾을 수 있도록 **주변 노드 발견**을 넣고, 이미지는 클라이언트에서 볼 수 있게 하되 저장·운영 책임은 별도 **media-node 프로그램**으로 분리한다.

---

## 1. 첫 진입 방식: 초대 링크 / QR / 카테고리 번들

공식 카테고리 목록을 만들면 중앙 운영자처럼 보일 수 있습니다.  
그렇다고 아무 진입 수단이 없으면 사용자가 처음에 들어올 수 없습니다.

따라서 첫 진입은 다음 방식으로 처리합니다.

```text
첫 사용자 진입 방식:
- 초대 링크
- QR 코드
- 카테고리 번들 파일
- 친구가 공유한 커뮤니티 번들
- 커뮤니티가 자체 배포하는 번들
```

카테고리 번들에는 다음 정보가 들어갑니다.

```text
카테고리 번들 내용:
- category_id
- 표시 이름
- 카테고리 헌장
- 텍스트 노드 목록
- 허용 미디어 노드 목록
- 관리자 공개키 목록
- 신고 정책
- 항소 정책
- 포크/이주 정보
```

사용자 화면 예시는 다음과 같습니다.

```text
친구가 보낸 game-korea 번들을 열었습니다.

이 번들은 다음을 추가합니다.

- game-korea 카테고리
- 텍스트 노드 3개
- 미디어 노드 2개
- 카테고리 헌장
- 관리자 공개키 목록

[추가] [자세히 보기] [취소]
```

---

## 2. 카테고리 이름 충돌 방지

중앙 카테고리 목록이 없으면 `game`, `news`, `politics` 같은 이름을 누가 차지하느냐가 문제가 됩니다.

따라서 카테고리는 단순 이름이 아니라 **내부 ID + 표시 이름**으로 나눕니다.

```text
내부 category_id:
생성자 공개키 + 카테고리 이름 + 생성 시각 + 헌장 해시

사용자에게 보이는 display_name:
game
```

같은 `game`이라는 표시 이름이 여러 개 있어도 됩니다.

```text
game
- game / 한국 게임 잡담
- game / 노정치 게임
- game / 고전게임
- game / 해외 게임 뉴스
```

이 구조는 나중에 포크, 이주, 연합 피드와도 잘 맞습니다.

## 3. 주변 노드 발견 기능

사용자 제안:

```text
현재 사용자들 중에 현재 노드 말고 어떤 노드에 많이 가입되어 있는지도 볼 수 있게 하기
```

이 기능은 공식 카테고리 목록 없이도 사람들이 새 커뮤니티를 찾을 수 있게 해주므로 필요합니다.

다만 그대로 만들면 사용자 추적처럼 보일 수 있습니다.  
따라서 최종 구조는 다음처럼 바꿉니다.

> 현재 노드/카테고리의 공개 참여자들이 함께 많이 쓰는 다른 노드를 **익명 집계**로 보여주는 기능

추천 기능명:

```text
- 주변 노드 발견
- 함께 이용하는 노드
- 관련 커뮤니티
- 이 카테고리 이용자들이 함께 보는 곳
```

추천 이름은 **주변 노드 발견**입니다.

---

## 4. 주변 노드 발견의 안전한 방식

절대 하면 안 되는 방식은 다음입니다.

```text
위험한 방식:
철수: A노드, B노드, C노드 가입
영희: B노드, D노드 가입
민수: A노드, 정치노드 가입
```

이렇게 하면 관심사, 정치 성향, 성적 취향, 지역, 소속 커뮤니티가 드러날 수 있습니다.

따라서 최종 방식은 다음 원칙을 따릅니다.

```text
안전한 방식:
- 개인별 가입 목록 공개 금지
- opt-in한 사용자 정보만 반영
- 익명 집계 통계만 표시
- 정확한 숫자보다 범위나 등급 표시
- 소수 인원 노드는 표시하지 않음
- 민감 카테고리는 기본 제외
```

화면 예시는 다음과 같습니다.

```text
주변 노드 발견

1. game-meme
   이 카테고리 장기 활동자들이 함께 많이 이용합니다.
   내가 신뢰한 사용자 중 일부도 사용 중입니다.
   [번들 보기] [추가]

2. retro-game
   관련 활동이 많은 커뮤니티입니다.
   [번들 보기] [추가]

3. meme-pack-node
   짤팩 공유용 미디어 노드입니다.
   [자세히 보기] [추가]
```

---

## 5. 주변 노드 발견은 opt-in 기반

사용자 구독 정보는 민감할 수 있으므로 기본 공개하면 안 됩니다.

설정은 다음과 같이 둡니다.

```text
[주변 노드 발견에 참여]

○ 참여하지 않음
○ 내가 직접 고른 노드만 공개 힌트로 제공
○ 내가 신뢰한 사용자에게만 공개
○ 익명 집계에만 사용

항상 제외:
☑ 민감 카테고리
☑ DM 관련 노드
☑ 숨김 처리한 노드
☑ 개인용 노드
```

기본값 후보:

```text
가장 안전:
참여하지 않음

현실적 기본값:
내가 직접 고른 노드만 공개 힌트로 제공
```

---

## 6. 주변 노드 발견의 조작 방지

스팸 세력이 가짜 계정 1,000개를 만들어 특정 노드를 인기 노드처럼 보이게 할 수 있습니다.

따라서 단순 가입자 수가 아니라 다음 요소를 반영합니다.

```text
반영 기준:
- 계정 생성 후 일정 기간 경과
- 해당 카테고리 활동 이력
- 장기 활동자 가중치
- 신고/스팸 이력
- 내가 신뢰한 사용자 가중치
- 신규 계정 대량 유입 감점
```

표시 예시는 다음과 같습니다.

```text
meme-node

- 공개 참여자 규모: 많음
- 장기 활동자 비율: 높음
- 내가 신뢰한 사용자: 일부 있음
- 신규 계정 급증 위험: 낮음
```

## 7. 클라이언트는 이미지 보기 지원

제품성을 위해 클라이언트는 이미지를 볼 수 있어야 합니다.

```text
클라이언트:
- 이미지 보기 가능
- 이미지 업로드 UI 제공
- media_ref 해석
- 썸네일 표시
- 이미지 자동 로드 설정
- 신뢰하지 않는 미디어 노드 차단
- 신고/숨김 버튼 제공
```

중요한 원칙:

> 클라이언트가 이미지를 보여줄 수는 있지만, 클라이언트가 자동으로 이미지 서버가 되면 안 됩니다.

이미지 저장과 공개 제공은 별도 미디어 노드가 담당합니다.

---

## 8. media-node 프로그램 분리

이미지는 클라이언트에서 볼 수 있게 하되, 특정 집단이나 개인이 쉽게 이미지 노드를 운영할 수 있는 프로그램을 따로 만듭니다.

최종 구조:

```text
client:
- 일반 사용자용
- 글/댓글/투표/이미지 보기
- 기본적으로 미디어 서버 기능 없음

text-node:
- 게시글, 댓글, 투표, 헌장 전파
- 이미지 원본 저장 안 함

media-node:
- 이미지 저장 전용 프로그램
- 개인/집단/카테고리가 선택적으로 운영

protocol:
- 공통 객체 스키마
- 서명, 해시, media_ref, tombstone 정의
```

배포물도 분리합니다.

```text
일반 사용자:
client만 설치

노드 운영자:
text-node 선택 설치

이미지 운영자:
media-node 선택 설치
```

---

## 9. media-node는 쉽게 운영 가능해야 함

이미지 노드 운영이 어려우면 아무도 운영하지 않습니다.  
그러면 이미지를 볼 수 있어도 커뮤니티가 제대로 굴러가지 않습니다.

media-node는 설치와 운영이 쉬워야 합니다.

```text
media-node 제공 방식:
- Docker Compose
- 단일 실행 파일
- 로컬 웹 대시보드
- 저장공간 제한 설정
- 카테고리별 허용 설정
- 신고 확인 화면
- 이미지 삭제/캐시삭제 버튼
- 로그 보관 기간 설정
```

대시보드 예시:

```text
미디어 노드 대시보드

상태:
- 저장공간: 12.3GB / 50GB
- 오늘 업로드: 231개
- 신고 대기: 7개
- 차단된 media_ref: 41개

정책:
- 최대 이미지 크기: 2MB
- 허용 형식: JPG, PNG, WebP
- 영상 업로드: 꺼짐
- EXIF 제거: 켜짐
- 기본 TTL: 30일
- 공개 검색: 꺼짐

카테고리:
- game-korea: 허용
- meme: 허용
- politics: 차단
```

## 10. media-node manifest

각 미디어 노드는 자기 정책을 공개해야 합니다.  
그래야 카테고리가 그 미디어 노드를 허용할지 판단할 수 있습니다.

```json
{
  "type": "media_node_manifest",
  "node_pubkey": "media-node-pubkey-1",
  "name": "game-meme-media-node",
  "operator_type": "individual_or_group",
  "accepted_media": ["image/webp", "image/jpeg", "image/png"],
  "max_file_size": "2MB",
  "video_allowed": false,
  "default_ttl": "30d",
  "exif_strip": true,
  "public_search": false,
  "report_supported": true,
  "cache_delete_supported": true,
  "allowed_categories": ["game-korea", "meme"],
  "policy_url": "optional",
  "compliance_statement": "operator_responsibility"
}
```

카테고리는 이 manifest를 보고 투표할 수 있습니다.

```text
이 카테고리에서 game-meme-media-node를 허용할까요?

- 이미지 최대 2MB
- 영상 금지
- TTL 30일
- 신고 기능 있음
- 캐시 삭제 지원
- 공개 검색 없음

[찬성] [반대]
```

## 11. 미디어 노드와 카테고리 관계

미디어 노드는 카테고리에 강제로 붙는 것이 아니라, 카테고리가 선택합니다.

```text
미디어 노드:
- 독립적으로 존재
- 개인이나 집단이 운영 가능
- 여러 카테고리에 이미지 제공 가능
- 자기 정책과 책임을 가짐

카테고리:
- 어떤 미디어 노드를 허용할지 결정
- 특정 미디어 노드를 차단 가능
- 특정 media_ref를 숨김 처리 가능
- 특정 짤팩을 허용/차단 가능
```

역할 구분:

```text
카테고리 관리자:
- 이 카테고리에서 무엇을 보여줄지 판단

미디어 노드 운영자:
- 이미지를 저장하고 제공하는 책임자
```

## 12. 이미지 표시 기본값

클라이언트는 이미지를 볼 수 있게 하되, 자동 로드는 조심해야 합니다.

권장 기본값:

```text
신뢰한 카테고리 + 허용 미디어 노드:
- 썸네일 자동 표시

처음 보는 미디어 노드:
- 흐림 처리
- 클릭해야 원본 로드

신고 누적 이미지:
- 기본 숨김

외부 이미지 링크:
- 자동 미리보기 없음
- 클릭해야 열림

민감 모드:
- 모든 이미지 기본 숨김
```

이렇게 하면 짤 문화는 살리면서도 위험한 이미지의 자동 확산은 줄일 수 있습니다.

## 13. 프로젝트 구조 보강

처음에는 모노레포 구조가 좋습니다.

```text
repo/
  apps/
    client/
    text-node/
    media-node/
  packages/
    protocol/
    crypto/
    governance/
    media-ref/
```

공통 프로토콜과 암호화, 거버넌스, media_ref 로직을 공유할 수 있습니다.

실제 배포는 분리합니다.

```text
client:
일반 사용자용

text-node:
텍스트 노드 운영자용

media-node:
이미지 노드 운영자용
```

## 14. 카테고리 번들 위조 방지

초대 링크, QR, 카테고리 번들 방식은 좋지만, 가짜 번들이 사용자를 피싱 노드나 악성 미디어 노드로 보낼 수 있습니다.

따라서 카테고리 번들에는 반드시 다음이 필요합니다.

```text
카테고리 번들에 추가할 것:
- category_id
- display_name
- 헌장 해시
- 생성자 공개키
- 현재 관리자 공개키 목록
- 텍스트 노드 목록
- 허용 미디어 노드 목록
- 번들 생성 시각
- 번들 만료 시각
- 번들 서명
- 이전 번들 해시
```

사용자 화면 예시:

```text
이 카테고리 번들은 다음 사람이 서명했습니다.

카테고리: game-korea
생성자 키: abcd...9123
헌장 지문: 9f12...aa77
포함된 텍스트 노드: 3개
포함된 미디어 노드: 2개
번들 만료일: 2026-08-01

주의: 이 번들은 공식 프로젝트가 추천한 것이 아닙니다.
[추가] [자세히 보기] [취소]
```

## 15. 키 분실·키 탈취 대응

익명 키 기반이면 중앙 계정 서버가 없어지는 장점이 있습니다.  
하지만 키를 잃어버리면 계정을 잃고, 관리자 키가 털리면 카테고리가 위험해집니다.

키 체계를 나눕니다.

```text
키 종류:
- 사용자 게시 키
- 투표 키
- 관리자 조치 키
- 복구 키
- 미디어 노드 운영 키
- 카테고리 헌장 서명 키
```

관리자 조치는 다중서명에 가깝게 둡니다.

```text
관리자 조치:
- 일반 숨김: 관리자 1명
- 장기 숨김: 관리자 2명 이상
- 헌장 변경: 투표 + 다중서명
- 관리자 교체: 투표 결과 서명
- 비상 복구: 장기 활동자 다중서명
```

키가 털렸을 때를 대비해 **키 폐기 증명서**가 필요합니다.

```text
revocation_certificate:
- 이 키는 더 이상 유효하지 않음
- 새 키는 이것임
- 서명자 목록
- 발급 시각
- 사유 코드
```

## 16. 안전 모드

특정 카테고리나 미디어 노드에서 문제가 터졌을 때, 클라이언트가 즉시 보수적으로 바뀌는 모드입니다.

```text
안전 모드가 켜지면:
- 모든 이미지 자동 로드 중지
- 처음 보는 미디어 노드 흐림 처리
- 신고 누적 콘텐츠 숨김
- 신규 노드 자동 추가 중지
- 외부 링크 미리보기 중지
- DM 첨부파일 차단
- 캐시 자동 삭제 주기 단축
```

카테고리 단위로도 가능해야 합니다.

```text
이 카테고리는 현재 안전 모드입니다.

사유:
- 미디어 노드 신고 급증
- 신규 계정 급증
- 관리자 키 의심
- 스팸 공격
```

## 17. media-node 기본값과 비상 모드

media-node는 처음부터 위험한 기본값을 피해야 합니다.

```text
media-node 기본값:
- 영상 업로드 off
- 파일 업로드 off
- 공개 검색 off
- 최대 이미지 2MB
- EXIF 제거 on
- TTL 30일
- 신고함 on
- 캐시 삭제 지원 on
- 저장공간 상한 필수
- 일일 업로드 제한 필수
- 신규 사용자 업로드 제한 on
- 외부 핫링크 제한 옵션
```

운영자 대시보드에는 **비상 정지 버튼**이 있어야 합니다.

```text
[비상 모드 켜기]
- 새 업로드 중단
- 기존 이미지는 썸네일만 제공
- 신고 누적 이미지는 즉시 숨김
- 외부 카테고리 요청 차단
```

## 18. 신고·숨김 사유 코드 표준화

“신고 가능”, “숨김 가능”만 있으면 카테고리마다 기준이 엉망이 됩니다.

공통 사유 코드를 둡니다.

```text
신고/숨김 사유 코드:
- spam
- flooding
- malware
- phishing
- doxxing
- harassment
- threat
- illegal_sexual_content
- csam_suspected
- non_consensual_intimate_media
- copyright_claim
- off_topic
- hate_or_abuse
- policy_violation
```

공개 로그 예시:

```text
조치 유형: temporary_hide
사유 코드: spam
조치자: moderator_pubkey
대상: post_hash
시각: 2026-06-22
만료: 24시간 후
항소 가능: 예
```

## 19. 공개 로그와 비공개 증거 분리

투명성은 필요하지만, 모든 증거를 공개하면 피해가 커질 수 있습니다.

따라서 로그를 두 층으로 나눕니다.

```text
공개 로그:
- 어떤 조치가 있었는지
- 누가 조치했는지
- 언제 조치했는지
- 사유 코드
- 항소 가능 여부
- 결과

비공개/제한 증거:
- 원본 이미지
- 피해자 정보
- 신고자 정보
- DM 신고 첨부물
- 개인정보 유출 내용
```

## 20. 주변 노드 발견의 민감 카테고리 보호

주변 노드 발견 기능은 유용하지만, 민감 카테고리에서는 위험합니다.

카테고리 헌장에 **discovery_policy**를 넣습니다.

```json
{
  "discovery_policy": {
    "allow_discovery": true,
    "allow_aggregate_stats": true,
    "min_k": 50,
    "show_exact_counts": false,
    "exclude_sensitive_tags": true,
    "allow_cross_category_recommendation": false
  }
}
```

카테고리 성격에 따라 다르게 설정합니다.

```text
게임/밈 카테고리:
- 주변 노드 발견 허용

정치/내부고발/지역 민감 카테고리:
- 발견 기능 기본 off

청소년/개인상담/소수자 카테고리:
- 발견 기능 매우 제한
```

## 21. 미러링 정책

미디어 노드가 꺼지면 이미지가 깨집니다.  
그렇다고 모든 노드가 자동 미러링하면 다시 책임이 퍼집니다.

따라서 다음 정책을 둡니다.

```text
자동 미러링:
- 기본 off

선택적 미러링:
- 카테고리 또는 미디어 노드가 명시적으로 허용한 경우만

미러링 조건:
- 신고 상태 정상
- 허용된 미디어 노드 간
- TTL 유지
- 원본 해시 유지
- mirror_of 필드 표시
```

복수 media_ref를 지원합니다.

```json
{
  "media_refs": [
    "media://nodeA/hash",
    "media://nodeB/hash",
    "media://nodeC/hash"
  ],
  "primary": "media://nodeA/hash"
}
```

## 22. 클라이언트 이미지 보안 샌드박스

이미지를 보여주는 순간 보안 문제가 생깁니다.

```text
클라이언트 이미지 보안:
- EXIF/메타데이터 제거된 썸네일 우선 표시
- 원본은 클릭 시 로드
- 이미지 디코딩 샌드박스

- 추적성 리소스 차단
- SVG 기본 금지
- 대용량 이미지 차단
- 애니메이션 GIF 제한 또는 WebP 변환
```

특히 SVG는 스크립트나 외부 리소스 문제가 생길 수 있으니 MVP에서는 제외합니다.

## 23. 프로토콜 버전·호환성 정책

분산 프로토콜은 한 번 퍼지면 바꾸기 어렵습니다.  
처음부터 버전 체계를 넣어야 합니다.

```text
protocol_version:
- 객체마다 포함
- 지원 최소 버전 표시
- 폐기 예정 필드 표시
- 호환 불가 변경은 major version
- 보안 취약 버전 경고
```

예시:

```json
{
  "type": "post",
  "protocol_version": "0.1.0",
  "min_client_version": "0.1.0",
  "created_at": "2026-06-22T00:00:00Z"
}
```

업데이트는 GitHub 확인만 한다면, 릴리즈 서명이 필요합니다.

```text
업데이트 보안:
- 릴리즈 파일 서명
- 체크섬 제공
- 서명 키 오프라인 보관
- 클라이언트가 서명 검증
- 자동 업데이트보다 수동 확인 우선
```

# 최종 요약

추가 보강의 핵심은 다음입니다.

```text
1. 공식 목록 없이도 초대 번들과 주변 노드 발견으로 진입성을 확보한다.
2. 주변 노드 발견은 개인 추적이 아니라 opt-in 익명 집계로만 한다.
3. 이미지는 클라이언트에서 보이게 하되, 저장·운영 책임은 별도 media-node로 분리한다.
4. media-node는 개인이나 집단이 쉽게 운영할 수 있게 한다.
5. 초대 번들, media-node, 신고, 로그, 이미지 표시에는 안전 기본값을 넣는다.
6. MVP에서는 영상, 파일, 공식 검색, 공식 게이트웨이, 공식 미디어 노드, 토큰을 제외한다.
```

최종 문장:

> **발견은 익명 집계로, 이미지는 분리형 미디어 노드로, 안전장치는 기본값으로.**
