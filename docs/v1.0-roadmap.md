# AgoraMesh V1.0 개발 로드맵 (최종안)

> 상태: 최종안 (Final Draft)  
> 목표: V1.0 첫 안정 릴리스까지의 개발 경로  
> 버전 정책:
> - `MVP-alpha` = Phase 0~2 (불변 원칙 동결 + 최소 P2P 텍스트 + 최소 UI)
> - `MVP-beta` = Phase 3~4 (헌장/관리자/신고 + media_ref/최소 media-node)
> - `v1.0` = Phase 5~10 (관리자 대시보드 + safe_mode + 번들 + 백업/이주 + 노드 패키징 + 보안 감사 + 릴리스)
>
> Phase 0에서 동결하는 것은 불변 원칙, 위협 모델, 책임 경계, 문서 구조뿐이다.  
> 객체 스키마, gossip 방식, 저장소 구조, 투표권 계산식, media_ref 세부 필드는 Phase 1~2를 거쳐 provisional에서 동결로 전환한다.

---

## 1. 기존 로드맵의 문제점 요약

1. **V1.0 범위 과대**: DM, 검색 인덱서, 웹 게이트웨이, 연합 피드 고급 기능, 짤팩 고급 기능까지 V1.0 필수로 포함되어 실제 개발 일정과 검증 부담이 비현실적이었다.
2. **MVP 개념 부재**: "MVP"라고 부르기에는 범위가 너무 컸고, MVP와 V1.0의 단계 구분이 없었다.
3. **Phase 0 과도 동결**: 프로토타입 구현 전에 gossip 방식, 저장소 구조, 투표권 계산식, media_ref 세부 필드, discovery 방식까지 frozen 처리하려 했다.
4. **UX 검증 시점 지연**: 클라이언트 UI와 관리자 대시보드를 후반 Phase에 배치해, 프로토콜 설계가 실제 사용성과 맞는지 초기에 검증할 수 없었다.
5. **시빌 방지 과도 추상화**: 전역 평판 그래프를 V1.0에서 구현하려 해 설명 가능성과 개발 현실성이 떨어졌다.
6. **공식 인프라 오해 가능성**: 릴레이/검색/게이트웨이 "기능 제공" 표현이 메인테이너가 운영하는 것처럼 읽힐 수 있었다.
7. **공개/비공개 증거 분리 미흡**: audit_log에 민감한 증거가 노출될 수 있는 구조가 명확히 분리되지 않았다.
8. **카테고리 상태 전환 규칙 부재**: `active`/`safe_mode`/`frozen`/`recovery`/`deprecated` 상태의 진입 주체, 조건, 지속 시간, 해제 조건이 정의되지 않았다.
9. **보안/테스트/운영 산출물 부족**: threat model, protocol conformance test, 릴리즈 서명, 백업/이주, 악용 시뮬레이션이 체계적으로 포함되지 않았다.
10. **착수 기준 모호**: Phase 0~1에 바로 들어갈 기준과 V1.0 전체를 시작하기 위한 기준이 구분되지 않았다.

## 2. 수정된 V1.0 정의

V1.0은 "완성형 분산 커뮤니티 전체"가 아니라, **일반 유저와 카테고리 관리자가 실제로 사용할 수 있는 최소 자치 텍스트 커뮤니티 제품**이다.

- 메인테이너는 프로토콜/클라이언트 개발자일 뿐, 어떤 인프라도 공식 운영하지 않는다.
- 텍스트 게시글·댓글·카테고리·헌장·선출형 관리자·신고·숨김·항소를 완전히 동작하게 한다.
- 이미지는 `media_ref`로 분리하고, 최소한의 `media-node` 레퍼런스 구현만 포함한다.
- 초반부터 최소 UI/TUI를 만들어 프로토콜 설계의 사용성을 검증한다.
- DM, 검색 인덱서, 웹 게이트웨이, 고급 연합 피드, 고급 짤팩은 V1.1+로 미룬다.

## 3. MVP / V1.0 단계 정의

| 단계 | Phase | 정의 | 사용자/관리자 기준 |
|---|---|---|---|
| **MVP-alpha** | 0~2 | 불변 원칙 동결 + 최소 P2P 텍스트 + 최소 UI/TUI | 개발자끼리 텍스트 글/댓글 주고받기 가능 |
| **MVP-beta** | 3~4 | 카테고리 헌장 + 관리자 + 신고/숨김/항소 + media_ref/최소 media-node | 한 카테고리에서 관리자 선출과 신고 처리 가능 |
| **v1.0** | 5~10 | 관리자 대시보드 + safe_mode + 번들 + 백업/이주 + 노드 운영자 패키징 + 보안 감사 + 릴리스 | 일반 유저/관리자가 실제 운영 환경에서 사용 가능 |

## 7. 각 Phase별 목표, 작업, 완료 조건

### Phase 0. 원칙 / 위협모델 / 문서 뼈대

**목표**: 구현 전 반드시 고정할 것만 고정하고, 나머지는 provisional로 둔다.

**고정 (frozen)**:
- 불변 원칙 (중앙 서버/릴레이/검색/게이트웨이/카테고리 목록/미디어 노드/추천 없음, 기본 접속 노드 내장 금지)
- 위협 모델
- 책임 경계 (메인테이너, 사용자, 카테고리 관리자, 텍스트 노드 운영자, 미디어 노드 운영자)
- 문서 구조
- 메인테이너가 운영하지 않는 항목 명시

**Provisional (이후 동결)**:
- 객체 스키마
- gossip 방식
- 저장소 구조
- 투표권 계산식
- `media_ref` 세부 필드
- discovery 방식

**작업**:
| # | 작업 | 산출물 |
|---|---|---|
| 0.1 | 불변 원칙 문서 작성 | `docs/protocol-overview.md` |
| 0.2 | 메인테이너 포지션 문서 작성 | `docs/maintainer-position.md` |
| 0.3 | 위협 모델 초안 작성 | `docs/threat-model.md` |
| 0.4 | 문서 구조 및 작성 규칙 확정 | `docs/README.md` 내부 링크 구조 |
| 0.5 | 법률 검토용 문구 확정 | `README.md`, `docs/legal-notes.md` |
| 0.6 | 카테고리 헌장 템플릿 초안 | `docs/charter-template.md` |
| 0.7 | 카테고리 번들 개념 초안 | `docs/category-bundle.md` |
| 0.8 | 릴리즈 서명 정책 초안 | `docs/release-signing.md` |
| 0.9 | discovery_policy 정의 | `docs/discovery.md` (정책 필드만, UI는 V1.1) |
| 0.10 | ADR 템플릿 및 첫 5개 ADR 초안 | `docs/adr/0001` ~ `0005` |

**완료 조건**:
- `docs/maintainer-position.md`가 메인테이너가 운영하지 않는 항목을 명시한다.
- `docs/threat-model.md`가 최소 5개 공격 시나리오를 포함한다.
- 문서 구조 목록이 확정되고 README에 링크된다.
- ADR 템플릿과 5개 초안 ADR이 작성된다.

### Phase 1. 최소 P2P 텍스트 프로토타입

**목표**: 서명된 텍스트 객체를 피어 간에 전파하고 검증한다.

**작업**:
| # | 작업 | 핵심 요구사항 |
|---|---|---|
| 1.1 | 키 기반 신원 | Ed25519 키페어, 키 폐기/갱신, `revocation_certificate` |
| 1.2 | `user_profile` 객체 | 표시명, 소개, 아바타 URL 금지, 서명 |
| 1.3 | `post` 객체 | 카테고리 참조, 텍스트, 타임스탬프, 서명, V1.0에서 immutable |
| 1.4 | `comment` 객체 | 부모 `post`/`comment` 참조, 스레드 트리, 서명, V1.0에서 immutable |
| 1.5 | `category` 객체 | `category_id`(creator_pubkey + display_name + created_at + charter_hash 기반 hash), display_name, 주제, 헌장 참조, 생성자 서명 |
| 1.6 | 메시지 교환 (provisional) | gossip 또는 직접 피어 연결, 객체 전파, 중복 제거. 초기 피어 연결은 수동 peer address, 로컬 discovery, 카테고리 번들 내 제3자 노드 후보만 사용. 기본 설정에 원격 공용 노드는 없음. |
| 1.7 | 로컬 저장소 (provisional) | SQLite/LevelDB 기반, 캐시 정책 |
| 1.8 | 기본 CLI 클라이언트 | 게시, 구독, 동기화, 조회 |

**완료 조건**:
- 두 대의 CLI 피어가 동일한 카테고리에서 게시글/댓글을 주고받을 수 있다.
- 3개 이상의 피어가 동기화되어 게시글/댓글을 일관되게 볼 수 있다.
- 서명 검증 실패 객체는 거부된다.
- 동일한 객체가 동일한 hash를 가진다.
- 클라이언트 재시작 후 로컬 상태가 복원된다.
- 중복 객체 전파가 제거된다.
- 미래 `created_at`을 가진 객체는 수신 거부 또는 경고 처리된다.
- 네트워크 단절 후 재연결 시 누락 객체가 동기화된다.
- 기본 설정에 원격 공용 피어가 포함되지 않는다.

### Phase 2. 최소 클라이언트 UI/TUI

**목표**: 프로토콜 설계가 실제 사용성과 맞는지 초기에 검증한다.

**작업**:
| # | 작업 | 핵심 요구사항 |
|---|---|---|

**작업**:

| 3.2 | `charter_amendment` 객체 | 헌장 개정안, 투표 연결, 2/3 이상 필요 |

| 3.9 | `appeal` 객체 | 항소 사유, 원래 조치 참조, 투표 연결, 결과 집행 |
| 3.10 | 공개 관리 로그 | `audit_log` 객체, 모든 관리자 조치 서명 기록 |
| 3.11 | 비공개 증거 분리 | `moderation_evidence_ref`로 민감한 증거는 audit_log에 노출하지 않음 |
| 3.12 | 투표권 계산 (provisional) | 활동 기간 기반, 1계정 1표 금지 방지, 전역 점수 없음, 개인정보 없음 |

**공개 로그 (audit_log) 포함 항목**:
- 조치 시각
- 조치 유형
- 사유 코드
- 대상 해시
- 조치자 공개키
- 만료 시각
- 항소 가능 여부
- `moderation_evidence_ref` hash 또는 encrypted reference (원본 콘텐츠는 포함하지 않음)

**비공개 증거 (분리 대상)**:
- 원본 이미지
- 신고자 정보
- 피해자 정보
- 개인정보 유출 내용
- 민감한 증거

**비공개 증거 전달 방식**:
- 비공개 증거는 전체 gossip에 포함되지 않는다.
- 증거는 active moderator key 목록에 대해 개별 암호화되어 해당 관리자에게만 전달된다.
- `audit_log`에는 원본 이미지, 신고자 정보, 피해자 정보가 절대 포함되지 않는다.
- `audit_log`에는 증거의 hash 또는 암호화된 참조만 기록하여 추후 대조가 가능하게 한다.
- 기본 보관 기간은 30일 또는 헌장 지정값이다.
- 관리자 해임 후 기존에 이미 복호화된 증거 회수는 best-effort로 처리한다.

**완료 조건**:
- 한 카테고리에서 관리자 선출 투표가 성사되고 결과가 `audit_log`에 남는다.
- 신고 5건이 같은 `post`에 쌓이면 `report_bundle`로 묶인다.
- 긴급 숨김 후 항소 절차가 자동 트리거된다.
- 오래된 헌장의 재전파가 방지된다.
- audit_log에 비공개 증거(원본 이미지, 신고자 정보 등)가 포함되지 않는다.
- `moderation_evidence_ref`가 active moderator key 목록에 대해 개별 암호화되어 전달되고, audit_log에는 hash/encrypted reference만 기록된다.
- 기본 보관 기간은 30일이며, 관리자 해임 후 복호화된 증거 회수는 best-effort다.
- `revocation_certificate` 이후에 생성된 객체는 거부되며, revocation 이전 객체는 보존되되 UI에서 폐기된 키로 작성된 객체임을 표시할 수 있다.

**카테고리 상태 전환 규칙**:

- **카테고리 상태 전환 규칙**:
  - 클라이언트 로컬 보호 모드(이미지 자동 로드 끔, 처음 보는 노드 흐림 등)는 클라이언트가 조건을 감지해 **자동으로 적용**할 수 있다.
  - 카테고리 공식 상태(`category_state`) 변경은 `category_state` 객체로 표현되며, 관리자 서명/다중서명/투표 중 헌장이 정한 절차를 거쳐야 한다.
  - safe_mode/recovery에 대한 클라이언트 로컬 보호는 자동 가능하지만, 공식 상태 변경은 항상 서명 객체로 기록된다.
  - `category_state` 객체는 `previous_state_hash`, `state_epoch`, `issued_at`, `expires_at`, `signer_pubkeys`, `required_quorum`, `charter_rule_ref`를 포함한다.
  - conflicting state(같은 `state_epoch`에 서로 다른 서명 집합으로 충돌하는 `category_state`)가 감지되면 클라이언트는 "상태 충돌" 경고를 표시한다.

| 상태 | 진입 주체 | 조건 | 자동/승인/투표 | 기본 지속 시간 | 해제 조건 | audit_log |
|---|---|---|---|---|---|---|
| `active` | 초기 | 카테고리 생성 완료 | - | 영구 | - | 카테고리 생성 시 기록 |
| `safe_mode` | 관리자 서명/다중서명/투표 | safe_mode 추천 조건 충족, 또는 긴급 상황 | 클라이언트 보호는 자동 가능, 공식 상태 변경은 관리자 승인/투표 | 24시간 또는 관리자 지정 | 조건 소멸 + 관리자 승인 또는 투표 | 진입/해제/조건 모두 기록 |
| `frozen` | 관리자 또는 투표 | 포크/이주 준비, 헌장 변경, 복구 모드 | 관리자 단독 24시간, 이후 투표 필요 | 관리자 지정 또는 투표 결정 | 투표 또는 관리자 합의 | 진입/해제 사유 기록 |
| `recovery` | 투표 | 장악 의심, 관리자 키 탈취, 대량 스팸 | 클라이언트 보호는 자동 가능, 공식 상태 변경은 투표 | 투표 결정 | 투표로 복구 완료 또는 포크 | 진입/투표/해제 기록 |
| `deprecated` | 투표 | 카테고리 폐쇄 결정 | 투표 | 영구 | 없음 (읽기 전용) | 폐쇄 사유 기록 |

**완료 조건**:
- 관리자가 10건의 신고가 묶인 카드에 대해 `[24시간 숨김]`을 누르면 `tombstone`이 생성된다.
- safe_mode 추천 조건을 만족하면 대시보드에 추천 배너가 표시된다.
- 카테고리 상태가 유저/관리자 UI 모두에서 이해하기 쉽게 설명된다.
- 상태 전환이 `audit_log`에 서명 객체로 기록된다.

## 9. MVP / V1.0 프로토콜 객체 목록

### 9.1 공통 객체 envelope

모든 프로토콜 객체는 다음 필드를 공통으로 포함한다.

| 필드 | 설명 |
|---|---|
| `type` | 객체 유형 (`post`, `comment`, `vote`, `report`, `category_bundle` 등) |
| `protocol_version` | AgoraMesh 프로토콜 메이저 버전 |
| `schema_version` | 해당 type의 스키마 버전 |
| `object_id` | canonical(`signing_payload`)의 hash (ADR 0001에서 canonical 직렬화 확정) |
| `created_at` | 생성 시각 (RFC 3339) |
| `author_pubkey` | 작성자 공개키 |
| `scope` / `category_id` | 객체가 속하는 카테고리 또는 범위 |
| `body` | 객체 실제 콘텐츠 |
| `signature` | `signing_payload` 전체를 `author_pubkey`에 대응하는 개인키로 서명 |

**signing_payload** (서명 대상):

```json
{
  "type": "...",
  "protocol_version": "...",
  "schema_version": "...",
  "created_at": "...",
  "author_pubkey": "...",
  "scope": "...",
  "body": "..."
}
```

- `signature`와 transport/local metadata는 signing_payload에 포함하지 않는다.
- `object_id`는 `canonical(signing_payload)`의 해시로 계산한다.
- canonical 직렬화 방식(바이트 순서, 키 정렬, 공백 처리 등)은 `docs/adr/0001-object-serialization.md`에서 확정한다.

> 확정 전. ADR에서 결정. 프로토콜과 클라이언트는 언어/런타임에 독립적이어야 한다.

| 영역 | 후보 | 비고 |
|---|---|---|
| 객체 직렬화 | Canonical JSON, CBOR | hash 일관성, 스키마 검증 |
| 서명 | Ed25519 | 키 길이, 속도, 생태계 |
| 해시 | SHA-256, BLAKE3 | 표준 vs 속도 |
| 저장소 | SQLite | 클라이언트/노드 공용, migration 용이 |
| 네트워크 | libp2p, WebSocket direct peer | NAT 우회 vs 단순함 |
| 클라이언트 | TUI 우선, 이후 Tauri GUI | 빠른 프로토타입 → 데스크톱 GUI |
| 스펙 문서 | JSON Schema + Markdown | 객체 스키마 동결용 |
| API 문서 | OpenAPI | media-node/텍스트 노드 인터페이스 |
| 미디어 처리 | ImageMagick/libvips, sharp | EXIF 제거, 썸네일 |
| 컨테이너 | Docker, Docker Compose | 제3자 노드 운영자 배포 |

## 13. 개발 착수 기준

### 13.1 Phase 0~1 착수 기준

1. `docs/maintainer-position.md`와 `docs/threat-model.md`가 확정된다.
2. `README.md`에 메인테이너가 운영하지 않는 항목과 법률 문구가 포함된다.
3. 문서 구조가 확정되고 최소한의 템플릿이 작성된다.
4. 기술 스택 후보(객체 직렬화, 서명, 해시, 저장소, 네트워크)가 1차 선정된다.
5. GitHub repository 구조와 CI 기반이 설정된다.

### 13.2 V1.0 전체 착수 기준

1. MVP-alpha(Phase 0~2)가 완료되어 텍스트 게시글/댓글을 UI로 주고받을 수 있다.
2. UX 리뷰를 통해 프로토콜 스키마 수정 사항이 반영되었다.
3. ADR 0001~0005가 provisional/accepted 상태로 작성되었다.
4. `docs/maintainer-position.md`가 메인테이너의 운영 불가 경계를 명확히 하고 있다.
5. 법률 검토용 문구가 `README.md`와 `docs/legal-notes.md`에 확정되었다.
6. 공식 서버/릴레이/검색/게이트웨이/카테고리 목록/미디어 노드를 내장하지 않음을 기술적으로 확인할 수 있다.

## 15. 기존 원칙 재확인

- 중앙 서버 없음
- 공식 릴레이 없음
- 공식 검색 인덱서 없음
- 공식 웹 게이트웨이 없음
- 공식 카테고리 목록 없음
- 공식 미디어 노드 없음
- 공식 추천 알고리즘 없음
- 기본 접속 노드 내장 금지
- 메인테이너는 프로토콜/클라이언트 개발자
- 카테고리는 헌장, 선출형 관리자, 신고, 숨김, 항소, 투표로 자치
- 이미지는 코어 프로토콜에 직접 넣지 않고 `media_ref`로 분리
- 텍스트 노드는 이미지 원본을 저장하지 않음
- 이미지 저장, 썸네일, 신고, 캐시삭제는 별도 media-node가 담당
- “삭제”보다 “숨김”, “전파 중단”, “캐시 삭제”, “로컬 표시 제한” 표현 사용
