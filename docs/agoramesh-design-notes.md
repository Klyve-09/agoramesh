## 1. 전체 결론

이 프로젝트는 **“관리자가 없는 커뮤니티”**가 아니라, **“중앙 운영자가 없는 분산 자치 커뮤니티 프로토콜”**로 잡아야 한다.

즉, 목표는 무정부 상태의 게시판을 만드는 것이 아니다. 목표는 중앙 서버, 공식 릴레이, 공식 검색 인덱서, 공식 게이트웨이, 공식 카테고리 목록 없이도 각 카테고리와 노드가 스스로 운영·관리·준법 대응할 수 있는 커뮤니티 인프라를 만드는 것이다.

핵심 방향은 다음과 같다.

> 중앙 운영자는 없다.  
> 하지만 각 카테고리에는 자체 헌장, 선출형 관리자, 신고, 긴급 숨김, 항소, 투표, 포크, 이주 절차가 있다.  
> 이미지는 없애지 않는다.  
> 대신 이미지 저장 책임을 코어 네트워크 전체에 뿌리지 않고, 카테고리별 선택형 미디어 노드로 분리한다.

---

## 2. 프로젝트 한 줄 정의

```text
중앙 서버 없이 작동하는 오픈소스 커뮤니티 프로토콜 + 클라이언트
```

사용자님은 커뮤니티 사이트 운영자가 아니라, 다음을 만드는 사람이다.

```text
- 오픈소스 클라이언트
- 공개 프로토콜
- 카테고리 생성 규칙
- 투표 시스템
- 신고/숨김/차단 도구
- 카테고리 헌장 구조
- 포크/이주 구조
- 선택형 미디어 노드 구조
- 노드 운영자 가이드
```

핵심은 **서비스 운영자**가 아니라 **프로토콜 개발자**로 남는 것이다.

---

## 3. 핵심 원칙

### 3.1 중앙 운영자 없음

프로젝트 메인테이너는 커뮤니티를 직접 운영하지 않는다.

```text
메인테이너가 하는 것:
- 클라이언트 개발
- 프로토콜 문서화
- 보안 업데이트
- 모더레이션 도구 제공
- 카테고리 헌장 템플릿 제공
- 노드 운영자 가이드 제공
```

```text
메인테이너가 하지 않는 것:
- 공식 서버 운영
- 공식 릴레이 운영
- 공식 검색 인덱스 운영
- 공식 게이트웨이 운영
- 공식 카테고리 목록 제공
- 공식 미디어 서버 운영
- 공식 신고센터 운영
- 공식 차단목록 배포
- 콘텐츠 합법성 최종 판단
```

### 3.2 관리자는 필요하다

관리자를 완전히 없애면 커뮤니티가 스팸, 도배, 분쟁, 불법 콘텐츠에 취약해진다. 문제는 관리자의 존재가 아니라 **누가 관리자를 임명하느냐**다.

```text
나쁜 구조:
중앙 운영자가 관리자를 임명함

좋은 구조:
카테고리 이용자들이 관리자를 선출함
```

### 3.3 이미지는 필요하지만 책임은 분리해야 한다

이미지, 짤, 밈, 캡처가 없으면 커뮤니티가 활성화되기 어렵다. 그러나 이미지를 모든 노드가 자동 저장하고 전파하는 구조는 위험하다.

따라서 이미지는 코어 프로토콜에서 분리하고, **카테고리별 선택형 미디어 노드**가 담당하도록 한다.

## 4. 절대 피해야 할 것

사용자님이 직접 아래 기능을 운영하면 다시 중앙 운영자처럼 보일 수 있다.

```text
- 공식 서버
- 공식 릴레이
- 공식 검색 인덱서
- 공식 웹 게이트웨이
- 공식 카테고리 목록
- 기본 접속 노드
- 공식 신고센터
- 공식 차단목록
- 공식 이미지 서버
- 공식 짤팩
- 인기글/추천글 중앙 운영
```

따라서 프로젝트의 기본 태도는 다음이어야 한다.

```text
나는 커뮤니티를 운영하지 않는다.
나는 커뮤니티를 만들 수 있는 도구와 프로토콜을 만든다.
```

## 5. 전체 구조

최종 구조는 5개 층으로 나누는 것이 좋다.

```text
1층: 코어 프로토콜
2층: 클라이언트
3층: 카테고리 거버넌스
4층: 노드/릴레이
5층: 선택형 미디어 노드
```

### 5.1 코어 프로토콜

```text
- 사용자 키
- 게시글
- 댓글
- 카테고리
- 헌장
- 투표
- 신고 메타데이터
- 숨김/차단 결정
- 포크/이주 정보
```

### 5.2 클라이언트

```text
- 글 읽기/쓰기
- 투표
- 신고
- 필터 적용
- 이미지 표시 여부 선택
- 카테고리 이주 안내
- 로컬 차단/숨김
```

### 5.3 카테고리

```text
- 자체 헌장
- 선출형 관리자
- 신고 처리
- 긴급 숨김
- 항소
- 규칙 변경 투표
- 포크/이주 절차
- 미디어 노드 허용 여부 결정
```

### 5.4 노드/릴레이

```text
- 게시글과 메타데이터 전달
- 자기 정책에 따른 저장/전파
- 자기 관할 법령 준수
```

### 5.5 미디어 노드

```text
- 이미지 저장
- 썸네일 생성
- 신고 처리
- 캐시 삭제
- 필요 시 필터링 시스템 운영
```

이 구조의 핵심은 **텍스트 커뮤니티 기능과 이미지 저장 책임을 분리하는 것**이다.

## 6. 카테고리 자치 구조

카테고리 관리자는 중앙 운영자의 직원이 아니라, 카테고리 내부에서 선출된 로컬 관리자다.

```text
카테고리 관리자:
- 카테고리 내부 투표로 선출
- 임기 있음
- 해임 가능
- 공개 로그 필수
- 긴급 숨김 가능
- 단독 영구 삭제 불가
- 항소 가능
- 권한 남용 시 재신임 투표
```

관리자의 권한은 “삭제 권력”이 아니라 **숨김, 태그, 필터, 임시 차단, 항소 처리**에 가깝게 제한해야 한다.

---

## 7. 투표와 즉시 조치의 분리

모든 것을 투표로 처리하면 운영이 너무 느려진다. 그래서 투표로 정할 것과 즉시 처리할 것을 분리해야 한다.

### 7.1 투표로 정할 것

```text
- 카테고리 규칙 변경
- 관리자 선출
- 관리자 해임
- 자동 필터 기준 변경
- 차단목록 채택 여부
- 항소 최종 판단
- 포크 여부
- 미디어 노드 허용 여부
```

### 7.2 즉시 처리할 것

```text
- 실시간 스팸
- 도배 계정
- 불법 의심 콘텐츠 임시 숨김
- 신고 큐 1차 정리
- 명백한 규칙 위반
- 위험한 미디어 노드 임시 차단
```

정리하면 다음과 같다.

```text
중요한 정책 결정 = 투표
긴급한 위험 대응 = 자동 필터 + 제한된 관리자 권한
```

## 8. 카테고리 헌장

카테고리는 단순히 이름만 있으면 안 된다. 반드시 헌장이 있어야 한다.

```text
카테고리 헌장 구성

1. 주제
2. 금지 행위
3. 관리자 수
4. 관리자 임기
5. 관리자 선출 방식
6. 관리자 해임 조건
7. 투표권 조건
8. 긴급 숨김 조건
9. 항소 절차
10. 관리자 부재 시 승계 방식
11. 이미지 허용 여부
12. 허용 미디어 노드 기준
13. 포크 및 이주 절차
```

헌장이 없으면 먼저 들어온 세력, 조직적으로 몰려온 세력, 목소리 큰 세력이 카테고리를 장악하기 쉽다.

## 9. 특정 세력 장악 방지

특정 세력이 카테고리를 먹는 것을 완전히 막기는 어렵다. 따라서 목표는 “완전 방지”가 아니라 **피해 제한**이다.

```text
목표:
- 먹기 어렵게 만든다.
- 먹어도 권한을 제한한다.
- 먹힌 카테고리에서 유저가 쉽게 빠져나오게 한다.
- 전체 네트워크는 못 먹게 한다.
```

필요한 장치는 다음과 같다.

```text
- 계정 1개 = 표 1개 금지
- 신규 계정 투표권 지연
- 장기 활동자 가중치
- 평판 기반 투표권
- 헌장 변경은 2/3 이상 필요
- 관리자 해임은 높은 기준 필요
- 긴급권한 남용 시 자동 신임투표
- 모든 관리 로그 공개
- 신규 계정 대량 유입 시 복구 모드
```

핵심은 특정 세력이 들어와도 전체 네트워크를 먹지 못하게 하고, 이용자들이 탈출할 수 있게 만드는 것이다.

## 10. 포크와 이주

카테고리가 먹혔다고 바로 포크하면 안 된다. 바로 포크하면 유저가 새 카테고리를 못 따라가고, 핵심 유저가 갈라지고, 라이트 유저는 이탈할 수 있다.

권장 흐름은 다음과 같다.

```text
장악 의심
→ 복구 모드
→ 거버넌스 변경 일시 동결
→ 관리자 재신임 투표
→ 장기 활동자 중심 비상 투표
→ 실패하면 포크
→ 클라이언트가 이주 안내
```

포크는 기본 운영 방식이 아니라 **최후의 탈출 장치**다.

### 10.1 소프트 포크

같은 카테고리 안에서 보기 방식만 나누는 방식이다.

```text
예:
- 기본 피드
- 노정치 필터
- 자유토론 필터
- 강한 규칙 필터
```

### 10.2 하드 포크

회복 불가능할 때 새 카테고리로 이주하는 방식이다.

```text
예:
- game-classic
- game-no-politics
- game-competitive
```

### 10.3 연합 피드

하드 포크 이후에도 연합 피드가 필요하다. 여러 포크가 생겨도 사용자는 `game alliance` 같은 연합 피드로 관련 카테고리를 묶어서 볼 수 있어야 한다.

```text
연합 피드 역할:
- 관련 카테고리 인기글 모아보기
- 공통 공지 공유
- 공통 이벤트 운영
- 공통 차단목록 선택 적용
- 여러 포크 동시 구독
```

## 11. 이미지와 미디어 노드

이미지는 빼면 안 된다. 짤, 캡처, 밈, 반응 이미지가 없으면 커뮤니티가 살아나기 어렵다.

그러나 이미지를 아래 방식으로 넣으면 위험하다.

```text
위험한 방식:
- 모든 노드가 이미지 원본을 자동 저장
- 모든 노드가 이미지 원본을 자동 전파
- 모든 카테고리 관리자가 이미지 필터링 시스템을 설치
- 게시물 본문과 이미지 원본이 같은 P2P 객체로 묶임
- 이미지가 한 번 퍼지면 사실상 제거 불가능
- 블록체인/IPFS에 이미지 원본 영구 저장
```

따라서 결론은 다음과 같다.

> 이미지는 넣되, 코어 프로토콜에서 분리한다.

```text
게시글/댓글/투표/헌장 = 코어 프로토콜
이미지 저장/썸네일/캐시삭제 = 선택형 미디어 노드
```

### 11.1 게시글 객체 예시

게시글 안에는 이미지 원본을 넣지 않는다. 게시글에는 `media_ref`만 넣는다.

```json
{
  "type": "post",
  "category": "game",
  "author": "user-pubkey",
  "text": "이 짤 뭐냐 ㅋㅋ",
  "media": [
    {
      "media_ref": "media://media-node-pubkey/hash",
      "hash": "sha256...",
      "thumb": "media://media-node-pubkey/thumb/hash",
      "declared_mime": "image/webp",
      "declared_size": 421337,
      "origin_node": "media-node-pubkey",
      "ttl": "30d"
    }
  ]
}
```

### 11.2 역할 분리

```text
일반 클라이언트:
- 이미지를 보여줌
- 자동 로드 여부 선택
- 신뢰하지 않는 미디어 노드 차단

텍스트 노드:
- 게시글, 댓글, 투표, 헌장 처리
- 이미지 원본 저장 안 함

카테고리 관리자:
- 특정 media_ref 숨김
- 특정 미디어 노드 차단
- 특정 짤팩 허용/차단
- 신고 누적 이미지 임시 숨김

미디어 노드:
- 이미지 업로드 수락
- 이미지 저장
- 썸네일 생성
- 신고 처리
- 캐시 삭제
- 자기 관할 법령 준수
```

카테고리 관리자는 이미지 저장소 운영자가 아니라, **그 카테고리 피드에서 무엇을 보여줄지 정하는 관리자**다.

## 12. 카테고리별 미디어 정책

미디어 노드는 중앙 프로젝트가 운영하지 않는다. 각 카테고리가 자기 헌장에 따라 미디어 사용 여부를 정한다.

```text
원칙:
- 미디어 노드는 중앙 프로젝트가 운영하지 않는다.
- 각 카테고리는 자기 헌장에 따라 미디어 사용 여부를 정한다.
- 미디어를 쓰는 카테고리는 신뢰할 미디어 노드를 선택하거나 자체 미디어 노드를 둘 수 있다.
- 미디어를 쓰지 않는 카테고리는 텍스트 전용으로 남을 수 있다.
```

### 12.1 카테고리별 예시

```text
/news
- 이미지 금지
- 외부 링크만 허용
- 자동 미리보기 없음

/game
- 신뢰 미디어 노드 3개 허용
- 짤팩 허용
- 이미지 최대 2MB
- 신고 5회 이상 자동 흐림

/meme
- 커뮤니티 미디어 노드 운영
- 짤팩 중심
- TTL 14일
- 영상 금지

/legal
- 텍스트 전용
- 첨부파일 금지

/free
- 아무 미디어 노드 참조 허용
- 단, 모든 이미지는 기본 흐림 처리
```

### 12.2 미디어 모드

```text
Mode 0. media_off
- 이미지 금지
- 텍스트 전용

Mode 1. external_link_only
- 외부 이미지 링크만 허용
- 자동 미리보기 없음

Mode 2. trusted_nodes_only
- 카테고리가 허용한 미디어 노드만 사용
- 기본 추천 방식

Mode 3. community_media_node
- 카테고리 또는 구성원이 자체 미디어 노드 운영
- 운영자와 책임 범위 명확해야 함

Mode 4. open_media_refs
- 아무 미디어 노드 참조나 허용
- 가장 자유롭지만 가장 위험
- 기본값으로는 비추천
```

기본 추천값은 **Mode 2: trusted_nodes_only**다.

## 14. 짤 문화 지원 기능

커뮤니티 활성화를 위해서는 단순 이미지 업로드보다 **짤팩**이 중요하다.

```text
짤팩:
- 자주 쓰는 반응 이미지 묶음
- 카테고리별 밈팩
- 사용자별 밈팩
- 노드별 밈팩
- 문제가 생긴 이미지만 팩에서 제거 가능
```

초기 이미지 제한은 이 정도가 좋다.

```text
- 이미지 최대 2MB
- JPG/PNG/WebP 허용
- 영상 업로드 금지
- 파일 업로드 금지
- EXIF 메타데이터 자동 제거
- 썸네일 생성
- 기본 TTL 30일
- 신고 누적 시 즉시 숨김
- 신뢰하지 않는 미디어 노드는 자동 로드 안 함
```

단, 프로젝트 메인테이너가 공식 짤팩을 만들면 중앙 큐레이션처럼 보일 수 있으므로, 공식 짤팩은 두지 않는 편이 좋다.

## 15. AI/DNA 필터링 논란에 대한 대응 방향

사용자님이 문제 삼는 것은 두 가지다.

```text
1. 비용 문제
중소 커뮤니티나 개인 운영자가 이미지 필터링 시스템, 장비, 솔루션, 운영 인력을 감당하기 어렵다.

2. 블랙박스 문제
어떤 이미지/영상 특징값이 DB에 들어갔는지 일반 이용자와 커뮤니티 관리자가 검증하기 어렵고, 오탐이나 악용 가능성이 우려된다.
```

이 프로젝트는 이 문제에 대해 “우회 도구”가 아니라 **책임 분리 구조**로 대응해야 한다.

```text
- 코어 프로토콜은 특정 중앙 DNA DB에 종속되지 않는다.
- 기본 프로토콜은 이미지 원본 저장을 강제하지 않는다.
- 이미지 저장은 선택형 미디어 노드가 담당한다.
- 미디어 노드 운영자는 자기 법령과 비용 판단에 따라 필터링 모듈을 선택한다.
- 카테고리 관리자는 이미지 저장 장비 운영자가 아니다.
- 프로젝트 메인테이너는 공식 필터 DB나 공식 차단목록을 운영하지 않는다.
```

즉, “AI 검열법 회피 앱”처럼 보이면 안 된다. **“중앙 서버 없는 커뮤니티 프로토콜이며, 미디어 책임을 노드 단위로 분리한다”**고 설명해야 한다.

## 16. 책임 분리

| 주체 | 역할 | 책임 방향 |
|---|---|---|
| 프로젝트 메인테이너 | 오픈소스 클라이언트·프로토콜 개발 | 공식 서버·검색·신고센터를 운영하지 않으면 낮음 |
| 사용자 | 게시, 댓글, 투표, 이미지 업로드 | 본인이 올린 콘텐츠 책임 |
| 카테고리 관리자 | 로컬 규칙 집행, 숨김, 항소 | 카테고리 관리 책임 |
| 텍스트 노드 운영자 | 게시글·댓글·투표 전파 | 저장·전파 정책 책임 |
| 미디어 노드 운영자 | 이미지 저장·공개 제공 | 이미지 관련 책임 큼 |
| 웹 게이트웨이 운영자 | P2P 콘텐츠를 웹으로 공개 | 플랫폼 운영자처럼 보일 수 있음 |
| 검색 인덱서 운영자 | 콘텐츠 검색 제공 | 검색·노출 책임 가능성 큼 |

최종 원칙은 다음과 같다.

```text
메인테이너는 도구를 만든다.
카테고리는 자치한다.
노드는 자기 책임으로 운영한다.
미디어 노드는 이미지 책임을 별도로 진다.
사용자는 어떤 노드와 필터를 따를지 선택한다.
```

## 17. 삭제 표현 주의

P2P 구조에서는 “삭제”라는 말을 함부로 쓰면 안 된다. 전 세계 모든 복제본을 지운다고 약속할 수 없기 때문이다.

### 17.1 피해야 할 표현

```text
- 전 세계 모든 복제본을 삭제합니다.
- 한 번 신고되면 완전히 사라집니다.
```

### 17.2 권장 표현

```text
- 이 카테고리 피드에서 숨김 처리됩니다.
- 이 노드는 해당 콘텐츠의 전파를 중단합니다.
- 이 클라이언트는 해당 콘텐츠를 표시하지 않습니다.
- 해당 media_ref는 이 카테고리의 차단목록에 추가됩니다.
- 로컬 캐시는 삭제됩니다.
- 이미 다른 노드에 복제된 사본은 각 노드 정책에 따릅니다.
```

관리자 권한도 “삭제 권한”보다 **숨김, 전파중단, 캐시삭제 권고, 카테고리 피드 제외**라고 표현하는 것이 좋다.

기본 금지:

- 불법촬영물

- 피싱

이런 금지 원칙을 둔다고 해서 곧바로 중앙 운영자가 되는 것은 아니다. 위험한 것은 금지 원칙이 아니라, 메인테이너가 모든 신고와 삭제 판단을 중앙에서 직접 맡는 구조다.

> 대신 이미지는 카테고리별 선택형 미디어 노드로 분리한다.  
> 메인테이너는 커뮤니티 운영자가 아니라 오픈소스 프로토콜과 클라이언트 개발자로 남는다.**

더 압축하면 다음과 같다.

> **“불법 콘텐츠를 못 지우는 무검열 이미지망”이 아니라,  
> “중앙 운영자 없이도 각 카테고리와 노드가 자치·신고·숨김·차단·이주·미디어 책임 분리를 할 수 있는 분산 커뮤니티 프로토콜”이다.**

### 피해야 할 표현

```text
- AI 검열법 회피 커뮤니티

- 삭제 불가능한 커뮤니티
- 운영자 책임 없는 익명 커뮤니티
- 불법 콘텐츠도 못 지우는 무검열망
```

### 권장 표현

```text
- 중앙 서버 없는 오픈소스 커뮤니티 프로토콜
- 중앙 플랫폼 의존도를 낮추는 커뮤니티 클라이언트
- 사용자가 직접 소유하고 관리하는 분산형 토론 네트워크
- 각 카테고리와 노드가 자기 관할 법령에 맞게 운영할 수 있도록 돕는 도구
- 신고, 숨김, 차단, 항소, 공개 로그를 지원하는 분산 커뮤니티 인프라
```

## 부록 B. 법률 관련 주의 문구

이 문서는 법률 자문이 아니라 프로젝트 설계 검토용 초안이다. 실제 출시, 배포, 노드 운영, 미디어 노드 운영, 게이트웨이 운영, 검색 인덱서 운영 전에는 대한민국 및 대상 국가의 법률 전문가 검토가 필요하다.
